## Git 版本控制实施方案

### 第一阶段：环境准备（预计10分钟）

#### 1.1 安装 Git
- 下载 Git for Windows（官网 git-scm.com）
- 安装过程中关键配置：
  - 选择代码编辑器（推荐 VS Code）
  - 选择 "Use Git from the command line and also from 3rd-party software"
  - 选择 "Use the OpenSSL library"
  - 选择 "Checkout Windows-style, commit Unix-style line endings"

#### 1.2 配置用户信息
```bash
git config --global user.name "你的GitHub用户名"
git config --global user.email "你的GitHub注册邮箱"
```

#### 1.3 生成 SSH 密钥（用于GitHub免登录）
```bash
ssh-keygen -t ed25519 -C "你的邮箱"
cat ~/.ssh/id_ed25519.pub
```
将公钥添加到 GitHub → Settings → SSH Keys

### 第二阶段：GitHub 仓库创建（预计5分钟）

1. 访问 GitHub.com，登录账号
2. 点击右上角 "+" → "New repository"
3. 填写：
   - Repository name: 五通道自动测试（或自定义名称）
   - Description: 五通道自动测试系统
   - 选择 Public 或 Private
4. 点击 "Create repository"
5. 记录仓库地址（两种方式）
   - HTTPS: https://github.com/用户名/仓库名.git
   - SSH: git@github.com:用户名/仓库名.git

### 第三阶段：本地仓库初始化（预计10分钟）

#### 3.1 初始化本地仓库
```bash
cd d:\JProgram\2026\五通道自动测试
git init
```

#### 3.2 创建 .gitignore 文件
创建文件排除不需要版本控制的文件：
- bin/、obj/（编译输出）
- .vs/（VS缓存）
- *.user、*.suo（用户配置）
- *.log（日志文件）

#### 3.3 首次提交
```bash
git add .
git commit -m "Initial commit: 初始化项目结构"
```

#### 3.4 关联远程仓库并推送
```bash
git remote add origin https://github.com/用户名/仓库名.git
git branch -M main
git push -u origin main
```

### 第四阶段：日常操作指南

#### 4.1 日常工作流程
```bash
# 每天开始工作
git pull origin main

# 编写代码...

# 阶段性保存（每完成一个功能）
git add .
git commit -m "描述你做了什么修改"

# 一天工作结束
git push origin main
```

#### 4.2 查看操作
```bash
# 查看当前状态
git status

# 查看提交历史
git log --oneline

# 查看文件差异
git diff

# 查看某文件的修改历史
git log -p 文件名
```

#### 4.3 分支操作
```bash
# 创建新分支
git checkout -b feature/新功能名

# 切换到分支
git checkout 分支名

# 查看所有分支
git branch

# 合并分支到主分支
git checkout main
git merge feature/新功能名

# 删除分支
git branch -d 分支名
```

### 第五阶段：协作开发流程

#### 5.1 协作准备
- 团队成员访问 GitHub 仓库
- 点击 "Fork" 复制仓库到自己的账号
- 每个成员有自己的 Fork 仓库

#### 5.2 协作工作流
1. 从官方仓库拉取最新代码
2. 创建功能分支进行开发
3. 提交修改到自己的 Fork
4. 在 GitHub 发起 Pull Request
5. 代码审核后合并到主仓库

### 注意事项

1. **敏感信息处理**：检查 Config/config.json 等配置文件，确保不提交密码、密钥
2. **提交粒度**：每次提交只包含相关的修改，便于追溯
3. **提交信息规范**：使用清晰的描述，如"修复通道切换bug"
